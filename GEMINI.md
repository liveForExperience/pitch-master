# GEMINI.md - 项目开发规约与上下文

## 1. 项目背景 (Context)
- **项目名称**: oldboy-club-manager
- **核心目标**: 构建一个业余足球爱好者管理平台，涵盖球赛管理、球队信息维护、球员录入、比赛分组自动分配以及赛后数据统计。
- **当前阶段**: 概念构思与基础架构搭建。

## 2. 技术栈 (Tech Stack)
- **核心框架**: Spring Boot (最新稳定版)
- **权限安全**: Apache Shiro
- **持久层**: MyBatis-Plus
- **数据库**: MySQL
- **版本管理**: Flyway (用于 SQL 脚本版本控制)
- **工具库**: Hutool (优先使用，禁止非必要地创建自定义工具类)
- **构建工具**: Maven

## 3. 编码标准 (Engineering Standards)
- **规范标准**: 严格遵循《阿里巴巴 Java 开发手册》。
- **注释原则**: 提倡代码“见名知意”，非逻辑极其复杂处不推荐使用冗余注释。
- **质量保证**: 每个功能模块必须配有单元测试 (JUnit 5 + Mockito)。
- **日志规范**: 必须包含必要的日志记录 (SLF4J + Logback)。
- **命名规范**: 变量名、方法名、类名必须语义明确，符合业务语境。

## 4. 协作指令 (Instructions for Gemini)
- **代码生成**: 在生成任何业务代码前，先确认是否符合 Flyway 脚本管理流程。
- **依赖引用**: 引入新依赖前必须征得用户同意。
- **任务执行**: 优先使用 Maven 命令进行构建 (`mvn clean package`) 和测试 (`mvn test`)。
- **逻辑实现**: 遇到通用的字符串、集合、IO 等操作，必须首选 Hutool 实现。

## 5. 核心算法设计 (Grouping Algorithm)
- **目标**: 将球员分配至 N 个小组（N 由赛事属性决定），确保整体竞技性平衡。
- **权重优先级**: 
  1. **实力均衡 (Skill Balance)**: 首要目标，基于球员综合评分。
  2. **位置互补 (Position Complementarity)**: 确保各组位置分布合理（如每组都有后卫）。
  3. **默契与意愿 (Chemistry & Willingness)**: 
     - **配合意愿**: 球员 A 对 B 的主观意愿 (0-1，未来扩展至 0-10)。
     - **默契度**: 基于历史共同参赛次数的统计值，优先让默契度高的球员同组。
- **扩展性**: 评分标准采用接口化设计，初始支持自评，预留第三方客观评价接口。

## 6. 待讨论模块 (Roadmap)
- **Phase 1 (MVP)**:
  - Shiro 用户登录、权限控制。
  - 球员档案（含技术分、位置、年龄、擅长脚）。
  - 赛事发布与人均费用自动计算。
  - **基础分组算法**: 基于单一技术分实现实力平均分配。
- **Phase 2 (Algorithm Plus)**:
  - 引入位置互补权重。
  - 球员关系模型（配合意愿、历史默契度）。
  - 分组策略切换功能。
- **Phase 3 (Statistics & Growth)**:
  - 赛后多维度数据录入（进球、助攻、MVP）。
  - 个人能力图表自动更新。
