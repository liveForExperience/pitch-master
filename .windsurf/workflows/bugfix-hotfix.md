# 缺陷修复 / 热修工作流

适用场景：

- 线上或本地复现的 bug
- 回归问题
- SQL / schema 不一致
- 接口路径不一致
- 状态流转错误

本工作流强调：**先找根因，再做最小安全修复**。

## 1. 先定义问题

先记录问题，而不是直接改代码：

```md
问题现象：
错误信息：
预期行为：
实际行为：
影响范围：
是否影响数据：
复现路径：
```

如果有报错信息，尽量保留完整异常、SQL、调用路径。

## 2. 先找根因，不先补症状

必须先检查以下一致性：

- `entity` 与数据库 schema 是否一致
- Controller 与 Service 的行为是否一致
- API 契约与前端调用是否一致
- 状态机假设是否被后端真正约束
- 文档与实现是否漂移

本项目高频根因类型：

- Flyway migration 缺失或不完整
- Java 实体新增字段，但数据库未补列
- 前端接口路径与后端 Controller 不一致
- Service 逻辑修改后未同步文档
- 状态机规则只写在前端，没有落到后端

## 3. 决定最小安全修复方案

优先选择最小、最直接、最可验证的修复方案。

例如：

- 补一个缺失的 migration
- 修正错误的字段映射
- 修复错误的接口路径
- 增加缺失的状态校验

如果问题涉及数据，还要判断是否需要：

- 只修 schema
- 补历史数据
- 补审计记录
- 预留回滚方案

## 4. 按安全顺序修复

推荐顺序：

1. 先修 schema / migration（如果有）
2. 再修后端
3. 若契约变化，再修前端
4. 最后同步文档

修复约束：

- 不混入无关重构
- 尽量保持兼容
- 修复范围尽量聚焦

## 5. 项目内建议验证方式

### 后端验证

- 运行 `mvn test`
- 如为 schema 问题，按项目方式运行 `com.bottomlord.Main`
- 确认 Flyway 正常执行
- 确认原报错 SQL / 接口不再失败

### 前端验证

如问题影响前端，在 `frontend/` 目录下执行：

- `npm run lint`
- `npm run build`

并手工验证原问题路径。

### 修复验证最低标准

- 能说明原问题根因
- 修复后原问题路径恢复正常
- 相邻核心流程未明显被破坏
- 不引入新的 schema 漂移

## 6. 把这类问题沉淀成规范

如果这个 bug 暴露出的是一类系统性风险，就同步更新文档或 workflow。

典型情况：

- schema 漂移：补 workflow 检查项
- API 不一致：补 `docs/API_SPEC.md`
- 数据模型变化：补 `docs/DATA_MODEL.md`

## 7. 提交前检查

提交前确认：

- 根因已明确
- 修复的是根因，不只是遮盖现象
- 已完成验证
- 文档已同步
- commit 信息可以直接看出修了什么

## 8. 建议提交信息

示例：

- `fix: 修复 match 表缺失 end_time 字段导致查询失败`
- `fix: 修正球员评分接口路径不一致问题`
- `fix: 补齐赛事状态回跳逻辑`
- `fix: 修复报名截止后仍可报名的问题`
