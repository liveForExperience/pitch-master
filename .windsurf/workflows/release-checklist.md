# 发布检查工作流

适用场景：

- 准备合并一批功能
- 准备发布一个重要改动
- 准备上线带有 schema 变化的版本

目标：保证本项目发布时**代码、数据库、接口、文档、验证**都一致。

## 1. 明确本次发布范围

先整理本次发布内容：

- 包含哪些功能
- 包含哪些修复
- 哪些未完成内容不进入本次发布
- 影响哪些模块
- 影响哪些文档
- 是否包含数据库 migration

可使用下面模板：

```md
本次发布内容：
影响模块：
数据库变更：
接口变更：
需要关注的风险：
发布后验证项：
```

## 2. 代码与 schema 就绪检查

发布前确认：

- 后端代码可启动或可通过最基本验证
- 前端构建通过
- 必要的 Flyway migration 已补齐
- `entity` 与数据库 schema 无漂移
- 前端调用与后端接口契约一致
- 无临时调试代码、无意外产物

建议执行：

### 后端

- `mvn test`
- 必要时按项目当前方式运行 `com.bottomlord.Main`

### 前端

在 `frontend/` 目录下执行：

- `npm run lint`
- `npm run build`

## 3. 功能回归检查

优先验证本项目的关键流程：

- 赛事发布与列表展示
- 报名与取消
- 分组与场次生成
- 比分录入与比赛过程
- 完赛与结算
- 涉及评分时，验证评分相关链路

如果本次只改一个局部模块，也至少验证它相邻的一条关键流程。

## 4. 文档同步检查

检查这些文档是否需要更新：

- `docs/API_SPEC.md`
- `docs/DATA_MODEL.md`
- `docs/ARCHITECTURE.md`
- `docs/WORKFLOW.md`
- `README.md`
- `TODO.md`

若代码与文档不一致，则本次发布视为未完成。

## 5. 仓库卫生检查

发布前确认：

- `git diff` 与预期一致
- commit 历史可读
- 无无关改动混入
- 无构建产物误提交

## 6. 部署准备

发布前至少确认：

- 环境假设清楚
- migration 执行顺序安全
- 出现问题时知道如何回滚或止损
- 发布后验证步骤明确

## 7. 发布后验证

发布或合并后，尽快验证：

- 应用是否正常启动
- 核心接口是否可用
- 关键页面是否可操作
- 是否出现数据库报错、控制台异常、字段缺失问题

## 8. 本项目特别注意事项

如果本次发布涉及以下内容，必须加严检查：

- **评分系统**：确认总分、三维评分、审计是否一致
- **赛事状态机**：确认状态推进与回跳逻辑正常
- **数据库字段变更**：确认 migration 已实际落库
- **前后端联调**：确认接口路径、字段名、枚举值一致
