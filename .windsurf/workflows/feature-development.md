# 功能开发工作流

适用场景：

- 新增功能
- 现有功能的中等及以上增强
- 涉及前后端联动、数据库结构调整、业务规则扩展的任务

本工作流面向当前项目：

- 后端：Spring Boot + MyBatis-Plus + Flyway + MySQL
- 前端：React + TypeScript + Vite + Ant Design / Ant Design Mobile
- 核心域：赛事生命周期、报名、分组、比赛过程、结算、评分系统

## 1. 先定义需求，不直接开写

先填写下面这段，避免边写边想：

```md
本次需求目标：
不做的范围：
影响模块：
是否涉及数据库：
是否涉及接口：
验收标准：
风险点：
```

至少回答清楚：

- 这个需求解决什么问题
- 明确不做什么
- 会影响哪些后端模块
- 会影响哪些前端页面或接口
- 是否需要新增字段、表、状态、审计
- 什么结果算完成

## 2. 识别影响面

编码前必须检查这些位置：

- 后端：`entity`、`mapper`、`service`、`controller`、DTO、Flyway migration
- 前端：`frontend/src/api`、类型定义、页面、组件
- 文档：`docs/API_SPEC.md`、`docs/DATA_MODEL.md`、`docs/ARCHITECTURE.md`、`docs/WORKFLOW.md`、`TODO.md`

如果需求命中以下高风险域，必须做更细的影响分析：

- 评分系统
- 赛事状态流转
- 报名与取消规则
- 费用结算
- 实时比分与 SSE

## 3. 先做最小设计

在开始实现前，至少明确以下几点：

- 数据结构怎么变
- 接口契约怎么变
- 状态流转是否变化
- 是否需要新增审计记录
- 对旧数据是否兼容

若涉及数据库变更，先想清楚：

- 是新增 migration，还是修正既有模型
- 应用启动时 Flyway 是否能自动落库
- 老数据是否需要兼容或回填

## 4. 按正确顺序实现

推荐固定顺序：

1. 先写或更新 Flyway migration
2. 再改后端领域模型和 Service 逻辑
3. 再确定接口请求/响应结构
4. 再改前端 `api` 层和页面
5. 最后同步文档

实现约束：

- Controller 只做编排，不堆业务判断
- Service 承载核心业务规则
- 页面层不硬编码核心业务约束
- 通用接口类型尽量集中管理
- 涉及数据库字段时，代码与 schema 必须同步

## 5. 项目内建议验证方式

### 后端验证

- 运行 `mvn test`
- 若涉及启动验证，按项目当前方式运行 `com.bottomlord.Main`
- 观察 Flyway 是否正常执行

### 前端验证

在 `frontend/` 目录下执行：

- `npm run lint`
- `npm run build`

若涉及页面交互，再手工验证关键链路。

### 功能验证最低标准

- 至少验证一个成功路径
- 至少验证一个失败或边界路径
- 若涉及前端，手工走通关键页面
- 若涉及 schema，确认应用启动无字段缺失或映射错误

### 核心域额外验证

若本次功能涉及以下内容，还要额外验证：

- **评分系统**：结算、衰减、管理员修正、审计
- **赛事流转**：发布、截止、分组、开赛、完赛、结算
- **报名规则**：报名、取消、No-show、豁免

## 6. 文档同步要求

只要接口、数据模型、状态机、评分规则发生变化，就同步更新文档。

通常需要检查：

- `docs/API_SPEC.md`
- `docs/DATA_MODEL.md`
- `docs/ARCHITECTURE.md`
- `docs/WORKFLOW.md`
- `TODO.md`

## 7. 提交前检查

提交前确认：

- 本地验证已经完成
- schema 与 entity 无漂移
- 关键流程已走通
- 文档已同步
- 本次提交主题单一、边界清楚

## 8. 建议提交信息

示例：

- `feat: 发布赛事页支持新的字段校验`
- `feat: 新增评分档案查询接口`
- `feat: 增加比赛结束时间字段与展示逻辑`
- `refactor: 收敛赛事状态流转逻辑`
